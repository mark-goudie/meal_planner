Three Approaches for the Meal Planner Redesign

Each builds on your existing Django codebase — the difference is how far we go

A

Targeted Fix — Polish What's There

Effort: ~2 weeks

Keep the current Django templates, fix the critical bugs (orphaned tables, migration issues, session-based AI flow), restructure CSS to be mobile-first, add ingredient normalization, and improve the meal plan week view to support drag-to-rearrange.

Pros

  • Fastest path to a working app
  • Minimal risk — incremental changes
  • Existing tests still valid

Cons

  • Django templates limit interactivity (no drag-drop, no swipe, no cooking mode without heavy JS)
  • Phone experience will always feel like a responsive website, not an app
  • Hard to add real-time features later (live shopping list sync between you and your wife)
B

Hybrid — Django API + Modern Frontend

Effort: ~4-5 weeks

Convert Django to a REST API backend (using Django REST Framework). Build a new mobile-first frontend with React or HTMX that feels app-like. Keep all existing models, services, and business logic — just replace the template layer with something designed for phones.

Pros

  • Phone-native interactions: swipe to rearrange meals, cooking mode, pull-to-refresh
  • Could become a PWA (installable on home screen, works offline)
  • Backend stays Django — your models, services, tests all survive
  • Clean API enables future mobile app or sharing features

Cons

  • More upfront work — need to build API layer + new frontend
  • Two codebases to maintain (backend + frontend)
  • Learning curve if unfamiliar with React/frontend tooling
C

Enhanced Django — HTMX + Alpine.js

Effort: ~3 weeks

Keep Django templates but supercharge them with HTMX (for dynamic updates without page reloads) and Alpine.js (for client-side interactivity). Rebuild the CSS mobile-first. Add structured ingredients, cooking mode, and draggable meal plans — all within Django's template system.

Pros

  • Single codebase — everything stays in Django
  • HTMX gives app-like feel (no page reloads) without a JS framework
  • Alpine.js handles cooking mode, drag-drop, swipe gestures
  • Existing services and models untouched
  • Can still become a PWA
  • Much simpler than React — Django developers can maintain it

Cons

  • Less interactive ceiling than React (complex real-time features harder)
  • HTMX community smaller than React — fewer ready-made components
  • If you later want a native mobile app, you'd still need an API

💡 My Recommendation: Option C

For a meal planner used by two people on their phones, HTMX + Alpine.js hits the sweet spot. You get app-like interactions (no page reloads, swipeable meals, cooking mode) without the complexity of a separate frontend. It keeps everything in one Django codebase, your existing backend is largely preserved, and HTMX is perfectly suited for the kind of interactions you need — updating a meal slot, ticking off shopping items, adding notes to a recipe.

If the app takes off and you want to commercialise, adding a DRF API layer later is straightforward — the services layer already exists.
